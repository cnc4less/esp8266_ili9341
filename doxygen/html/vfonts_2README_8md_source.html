1 font_to_svg
2 ===========
3 
4 This code will convert a single character in a TrueType(R) font file
5 into an SVG 'path' shape.
6 
7 ### basic usage:
8 
9 Let's say you have a TrueType font file, such as the popular
10 FreeSans.ttf. Let's pull out the path information for the letter "B" and
11 transform it into SVG. A simple C++ program code like the following can
12 do that.
13 
14  #include <font_to_svg.hpp>
15  int main()
16  {
17  font_to_svg::glyph g( "FreeSans.ttf", 66 );
18  std::cout << g.outline();
19  }
20 
21 Compile and run, and the program will generate an SVG path out of the
22 TrueType information for the character of Unicode U+0066 (the Latin
23 letter B), something like this:
24 
25  <svg>
26  <path d='M 591,-180 Q 591,-150 580,-121 Q 569,-92 543,-63 ... Z />
27  </svg>
28 
29 'M' is 'move to', and 'Q' is for Bezier curves. Open the resulting .svg
30 file in a web browser and you get something like this:
31 
32 ![Rendering of font glyph](screenshot.png " The Letter B ")
33 
34 Another example: Floral Heart, Unicode 0x2766:
35 
36 ![Rendering of font glyph](screenshot2.png " Floral Heart ")

38 ### Current Status
39 
40 This project has some bugs but will handle 'standard' fonts reasonably well.
41 
42 Currently this project consists of a C++ language header that can used
43 in conjuction with the Freetype library to create a basic conversion
44 program that will extract a single character from a .ttf file and create
45 a matching .svg file.
46 
47 The program is stable for basic use. It can output plain glyph shapes or
48 it can also output 'debugging' information like the node points & lines.
49 There are some bugs with bounding boxes and other 'typographic box'
50 issues like Bearing. Also calculation of the SVG "g" tag has some issues
51 with transforms/footers.
52 
53 The code does not currently support OpenType or it's features, such as
54 ligatures. It does not support creating an "SVG Font". It only does very
55 basic conversion of Truetype glyphs to SVG path shapes. It might not
56 be useful for web fonts or other usages.

58 ### More example programs
59 
60 There are several example programs included. They use the cmake build
61 system ( http://www.cmake.org ). For example:
62 
63  cmake .
64  make
65  ./example1 `locate tahoma.ttf` 66 > out.svg
66  firefox out.svg
67 
68 Example 1 and Example 2 just do basic output of a single character into
69 a .svg path. The first argument is a truetype font filename, and the
70 second argument is a Unicode codepoint number. For example:
71 
72  ./example2 ./FreeSerif.ttf 66 > example2.svg
73 
74 In the line above, 66 = unicode/ascii for the letter 'B'. You can use
75 hexadecimal numbers as well. For example, the Cherokee Letter O has
76 hexadecimal Unicode point 0x13A3:
77 
78  ./example1 ./FreeSerif.ttf 0x13A3 > example2.cho.svg
79 
80 To view the resulting svg file, it is easiest to use a browswer like
81 Firefox. Go to 'file/open' and point it to your output file. Or type
82 something like 'file:///tmp/x.svg' into your address bar. You can also
83 get the Firefox addon that can auto-reload files when they change on
84 disk.
85 
86 Another option is to use svg view/edit programs like Inkscape.
87 
88 Example 3 was created by github user Ebraminio to generate multiple .svg
89 files from a single GPL font of ancient Persian letters ( Xerxes.ttf,
90 available by a web search )

92 ### Detail on using in your own project
93 
94 As noted, font_to_svg is a 'header library' so you dont need to
95 compile any libraries just include the header and use it. You will still
96 need to link to Freetype however, using your build system.
97 
98 Freetype's website is here: http://www.freetype.org/
99 
100 font_to_svg uses freetype to deal with vaguaries and variations of
101 Truetype file formats. font_to_svg does not use any of Freetype's bitmap
102 font-rendering code. font_to_svg is a pure "outline curve" renderer to be
103 used for vector/curve based output.
104 
105 Font_to_svg comes with a very permissive BSD style copyright license
106 described in the LICENSE file in this distribution. Acknowledgment in
107 the product documentation would be appreciated but is not required.
108 
109 To understand the usage, look at the files named 'example*.cpp' that come
110 with the source code. You can output the bare svg path data, or a bunch
111 of debugging information.
112 
113 Don't forget that some fonts have restrictive copyright licenses that
114 forbid this type of conversion. You can use fonts from
115 http://openfontlibrary.org If you want to be safe from accusations of
116 copyright violation.

122 ### How does Truetype actually work then? How does this program work?
123 
124 Truetype fonts consist of a group of characters layed out in a file. The
125 truetype file format is very complicated, with various issues related to
126 character sets, glyph indexes, duplicated positions, etc etc etc. font_to_svg
127 uses the Freetype library to make it easy to deal with those problems
128 and to load the file into memory.
129 
130 Each character in the Truetype file has a sequence of 'contours', which
131 are basically a sequence of curves and lines that form a single shape.
132 Some contours count as the 'body' of the letter, while others are the
133 'holes' of a letter. The letter 'B' for example has a single contour for
134 the 'outside' shape, and two 'hole' contours for the two inside shapes.
135 
136 Truetype marks these contours in a special way - by the 'clockwise' ness
137 of the order in which it lists the points that make up a contour. The
138 contours that have their points in 'clockwise' order are considered
139 'solid' The contours that have their points listed in 'counter
140 clockwise' (also called 'anti-clockwise') order are considered 'holes'.
141 For example, imagine the letter 'o'. The points making up the 'outside'
142 contour are listed in clockwise order, while the points making up the
143 'inside' contour are listed in counter-clockwise order.
144 
145 But what type of points make up a contour? They are simple 2-dimensional
146 Cartesian points with integer coordinates. The points for a contour are
147 listed in order in the Truetype file. The curves and lines of a contour
148 are drawn between the points using the rules of Bezier Curves. This
149 means there are actually two different types of points - 'on the curve'
150 points and 'off the curve' points. The 'on the curve' points actually
151 have lines or curves coming out of them and/or touching them. 'Off the curve'
152 curves represent the 'control' points between the 'on the curve' points.
153 
154 Now, Truetype doesn't just use any old Bezier curve. It uses a type of
155 Bezier called 'Quadratic', which consists of two endpoints 'on the
156 curve' and one 'control point'. If you check the Wikipedia article for
157 Bezier Curve, http://en.wikipedia.org/wiki/B%C3%A9zier_curve , you can see simple animated examples.
158 
159 In this example, p0 is 'on the curve', p2 is also 'on the curve', but
160 p1 is 'off the curve'. In other words, p1 is a 'control point'.
161 
162 ![Bezier.gif](bezier.gif " The Letter B ")
163 
164 The math is not actually as complicated as it looks on the wiki page-
165 you can draw a Bezier Quadratic using only a ruler and a pencil. I.E.
166 you can actually do it without any transcendental functions or
167 complicated algebra, just using simple ratios and integer math. But I
168 digress.
169 
170 Now... in Truetype there is a special feature in the way off-curve
171 points are listed. If two are listed in a row, then there is actually a
172 'hidden' or 'implied' on-curve point at a position exactly halfway
173 between them. font_to_svg does not draw these points but they are easy to imagine
174 when looking at the debug code.
175 
176 Now. How does font_to_svg do SVG output? It basically just copies the point
177 and contour information in the TrueType file and splits it into SVG
178 paths. The actual rendering of the SVG file to something on a computer
179 screen is left to the SVG rendering programs, like web browsers or
180 Inkscape. The most complicated thing that these renderers do is probably
181 the "non-zero winding rule", which is a geometry rule that determines
182 how to properly draw the 'holes' and 'body' contours of a character
183 glyph. font_to_svg doesn't have to worry about that either - it just inserts
184 the svg property 'non-zero' fill-type into the generated SVG and the SVG
185 renderer should take care of the rest.
186

191  * FType + contours: http://www.freetype.org/freetype2/docs/glyphs/glyphs-6.html
192  * TType contours: https://developer.apple.com/fonts/TTRefMan/RM01/Chap1.html

198 ### What about rendering multiple characters?
199 
200 To do this properly for all languages, you actually need a 'layout
201 engine', like Harfbuzz and/or Pango. These are at present, far beyond
202 the scope of this project.
203 
204 It may be possible to render strings of characters in 'non-connected'
205 scripts, for example English ( Latin alphabet), or Chinese, using the
206 basic bounding box information. However this library, at present, is
207 probably not very suitable for ligature + combination scripts like
208 Arabic or Devanagari.
